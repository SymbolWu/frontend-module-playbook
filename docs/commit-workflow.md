# 仓库提交工作流

## 目标

约束这个文档项目自身的提交信息风格和提交流程，让仓库历史能直接表达本次变更的意图。

## 提交类型

本仓库使用以下提交类型前缀：

- `NEW:` 新增全新内容或能力
- `IMPROVE:` 对现有内容做增强、重构或结构优化
- `FIX:` 修复错误、错误文案、错误链接或错误规则
- `DOC:` 纯文档说明类调整
- `RELEASE:` 发布版本、补 release note、调整版本信息
- `TEST:` 与验证、测试或校验机制相关的调整
- `BREAKING:` 引入破坏性变更，影响旧路径、旧结构或旧使用方式

## 提交信息格式

推荐格式：`<TYPE>: <简短描述>`

额外约束：

- 类型前缀保留英文大写，如 `NEW:`、`DOC:`、`BREAKING:`
- 冒号后的描述必须使用中文
- 描述应直接说明动作和对象，不使用英文句子作提交正文标题

例如：

- `NEW: 新增公开 skill 仓库结构`
- `IMPROVE: 优化面向公开分发的 README`
- `FIX: 修正 AGENTS 中的 skill 引用路径`
- `DOC: 补充目标项目引入模板`
- `RELEASE: 发布 v0.1.0`
- `BREAKING: 删除重复的 workflow 文档副本`

## 选择类型的规则

- 新增一类全新文档、目录或能力时，优先用 `NEW:`
- 已有内容的整理、增强、重写或结构优化，优先用 `IMPROVE:`
- 纯修正错误时，优先用 `FIX:`
- 只改仓库说明、README、接入文案、发布说明时，优先用 `DOC:`
- 版本号、release note、发布动作相关，优先用 `RELEASE:`
- 如果本次改动会改变已有用户的接入路径或目录结构，应优先评估是否用 `BREAKING:`
- 即使类型前缀为英文，后面的提交描述也必须保持中文。

## 提交前检查

- 先确认改动是否仍然聚焦于一个主要意图
- 先确认没有误带入无关文件
- 先确认 README、AGENTS、docs 与 skill 正文的路径引用保持一致
- 如果改了 skill 结构，确认 `SKILL.md` 与 `references/` 的引用仍然有效

## 当前仓库的推荐做法

- 一个提交尽量只表达一个明确动作
- 不要把“目录重构 + README 重写 + 发布版本”全部混在同一个提交里，除非它们天然属于同一发布动作
- 如果只是调整文案或说明，不要误用 `NEW:` 或 `BREAKING:`
