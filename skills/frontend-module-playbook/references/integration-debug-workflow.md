# 联调排查工作流

## 目标

沉淀一套前后端联调中的有效排查顺序，尤其针对“上传成功但后续失败”“任务创建成功但状态不推进”“报告或详情页打不开”这类链路问题。

## 联调前先确认

- 前端当前是否已切到正确的 backend 模式。
- API Base URL 是否指向正确服务。
- 环境变量是否通过统一入口收口，而不是在多个局部文件里散落拼接。
- 关键上下文参数是否能通过 URL 透传并恢复页面状态。

## 推荐排查顺序

### 1. 先看服务是否可达

- 优先检查健康接口。
- 如果健康检查失败，先判断是网络、环境变量还是服务本身不可用。

### 2. 再看输入链路是否成立

- 上传接口是否真的返回了后续流程需要的标识。
- 前端是否真的把后端返回的 ID 用在后续请求里，而不是仍然沿用本地文件名或临时状态。

### 3. 再看任务创建

- 创建任务接口是否成功返回任务标识。
- 输入模式、附件 ID、选项参数是否与当前交互方式一致。

### 4. 再看任务轮询

- 页面是否真的还在轮询。
- 接口返回的 `status`、`stage`、`progress` 是否持续推进。
- 如果长时间停在某个阶段，要先判断是前端没继续轮询，还是后端任务真的卡住。

### 5. 再看结果拉取

- 只有在任务完成后再拉结果。
- 结果接口失败时，先区分是任务未完成、结果未生成，还是任务 ID 本身无效。
- 结果页直开时，要确认页面能只依赖 URL 参数拉取结果，而不是强依赖内存态。

## 上下文参数调试经验

- 任何涉及复现场景、截图排查、跨人协作的问题，优先保留任务 ID、阶段、状态等关键上下文。
- 当 URL 参数和内存态都存在时，调试场景通常优先以 URL 参数为准。

## 常见问题定位方式

### 上传看起来成功，但提交失败

- 检查前端是否保存了后端返回的真实 ID。
- 检查请求参数是否与当前输入方式匹配。

### 页面一直转圈

- 检查页面是否仍在轮询。
- 检查接口返回的进度和阶段是否推进。
- 检查异常路径是否把错误静默吞掉了。

### 结果页打不开

- 检查任务是否已完成。
- 检查结果接口是否按任务 ID 请求。
- 检查页面是否错误依赖了旧的内存态。

## 工作方式

- 先用最小可验证接口缩小问题范围，不要一开始就假设整个链路都坏了。
- 先确认是哪一段断了：上传、建任务、轮询、结果、页面回退。
- 如果后端脚本验证与前端表现不一致，要分别记录接口结果和页面行为，不要混为一个结论。
