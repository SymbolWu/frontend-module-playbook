# 评审工作流

## 目标

约束开发者或 AI 在代码评审时，优先识别真实风险、行为回归、边界遗漏和测试缺口，而不是只做表面风格检查。

## 评审前先看什么

- 先确认本次改动是否仍限制在目标模块或其必要接入范围内。
- 先判断这次改动属于哪一层：页面组件、ViewModel、transformer、service、repository、mock、文档。
- 先分清它改的是视觉表现、交互流程、状态管理还是后端契约映射，不要混在一起审。

## 优先检查顺序

### 1. 流程完整性

- 页面之间的跳转是否仍符合既定链路。
- 是否存在只改进入路径、没改返回路径的情况。
- 是否存在只改读取方式、没改写入方式的情况。

### 2. 状态与数据来源

- 中间态是否被正确写入、读取、清空。
- URL 参数、内存态、mock 数据、后端返回之间是否存在优先级冲突。
- 是否出现页面直接消费原始后端 DTO、绕过 transformer 的情况。

### 3. 分层边界

- 是否把页面专属逻辑错误地下沉到共享层，或把共享逻辑散回页面层。
- 是否在组件中堆积过多流程控制和业务判断。
- 是否把展示逻辑写进 service 或 repository。

### 4. 错误处理与降级

- 查询失败、无数据、缺参、任务失败时是否有一致的回退路径。
- 是否存在只有 happy path 正常、异常路径断裂的情况。
- 用户直接打开 URL 或恢复页面上下文时，页面是否能正确降级而不是白屏。

### 5. 约束一致性

- 是否仍优先使用现有 UI 体系，而不是重复造通用控件。
- 是否与当前模块已有注释风格、命名风格、提交规范冲突。

## 怎么提问题

- 先报真正影响行为、数据正确性、流程闭环的问题。
- 问题描述要明确指出“现象 + 原因 + 影响”。
- 尽量指到具体文件和位置，而不是笼统说“这里有问题”。
- 如果是产品规则未确认导致的风险，要写清这是待确认项，而不是直接判定为 bug。

## 输出顺序

- 先列 findings，按严重程度排序。
- 再列 open questions / assumptions。
- 最后才给简短总结或变更概览。
