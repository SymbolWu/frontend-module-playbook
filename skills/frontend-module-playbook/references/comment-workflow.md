# 注释工作流

## 目标

约束开发者或 AI 在补充、整理和统一代码注释时的风格与覆盖重点，让注释既能帮助人类理解，也能帮助后续 AI 快速接手。

## 总原则

- 注释优先解释设计原因、职责边界、状态流转和调用语义。
- 注释不能只复述代码表面动作。
- 注释密度应尽量连续，不要一段写得很细、相邻关键逻辑完全无说明。
- 不要为了凑注释而给每一行写解释。
- 当纯逻辑文件需要按职责分段时，可配合 `#region` / `#endregion` 组织阅读。
- `region` 标题统一使用中文，并直接表达区块职责。

## region 使用原则

- `region` 适用于 transformer、service、repository、复杂工具函数文件等纯逻辑文件。
- `region` 的作用是帮助阅读，不是替代拆分。
- 当一个文件需要依赖大量 region 才能勉强读懂时，应优先拆成多个文件。
- 组件 TSX 文件不把 `region` 当作默认解法；组件过长时优先拆组件、拆目录、拆辅助文件。

## 类型文件注释要求

- 接口、类型别名、DTO、错误对象、service 抽象要尽量补齐明确注释。
- 当类型承担接口契约作用时，优先补充到字段级注释，而不是只写一句笼统说明。
- 字段注释应说明它代表什么、由谁提供、在哪个流程被消费。

## ViewModel 注释要求

- React hook 声明也应补充注释，优先说明状态保存的是什么、来源是什么、为什么在这一层维护。
- 优先解释状态来源优先级。
- 优先解释页面跳转时为什么要写入或读取 repository。
- 优先解释轮询成功、失败、回退的原因。
- 从 ViewModel 中提取出来的纯函数或副作用辅助函数，也应保留职责注释，说明它收敛了什么规则、依赖哪些入参、服务哪段流程。
- 当页面级 ViewModel 收敛为 `view-model/` 子目录后，主 hook 文件与 `helpers.ts` 仍应分别补充职责说明，避免目录拆分后语义反而变散。

## 组件注释要求

- 导出的主组件应优先补职责注释，说明该组件服务的页面、模块或业务区块。
- 当前文件内的辅助方法也应尽量补注释，说明它负责格式化什么、分流什么、收敛什么展示规则。
- 当页面目录拆出 `components/` 后，每个内部组件文件也应有一句职责说明，明确它负责的是“顶部摘要区”“规则区”“导出弹窗”这类业务区块，而不是只写成宽泛的“局部组件”。

## Transformer / Service 注释要求

- 导出映射方法要说明“从哪种数据结构映射到哪种数据结构”，以及它服务的页面或流程。
- 私有辅助方法要说明它收敛了什么规则，例如展示名回退、状态折叠、证据提取、默认值兜底。
- 公开请求方法要写清它请求什么、返回给谁用、服务于哪一段流程。
- 底层适配方法要说明为什么这一层要收敛字段、转换错误或隐藏后端细节。
- 如果文件按 `region` 组织，区块标题也要直接表达职责，例如“公开入口”“字段映射”“错误兜底”，不要写空泛标题。

## 纯逻辑文件注释要求

- 除 transformer 外，`services/`、`repository.ts`、纯工具函数文件等不直接承载 JSX 的逻辑文件，也优先遵循中文注释与按职责分区的规则。
- 当一个逻辑文件方法较多、职责可分段时，可使用中文 `region` 把“公开入口”“状态收敛”“存储读写”“错误兜底”等逻辑分开。
- 导出方法优先解释它对外提供什么能力、被哪段流程消费；私有方法优先解释为什么需要单独收敛这一步。
- 不要为了凑注释而给每一行写说明；重点是让后续维护者快速理解每个逻辑区块的职责和边界。

## 不推荐写法

- 给变量赋值
- 渲染一个按钮
- 这里调用接口
- 只解释语法，不解释业务含义

## 推荐写法

- 解释为什么某个上下文以 URL 参数为准而不是以内存态为准。
- 解释为什么返回首页前要清空草稿和分析态。
- 解释为什么某个 DTO 字段只在成功态或失败态出现。
- 解释为什么一个辅助函数必须放在 ViewModel 层，而不是放在组件或 service 层。
