# 实现工作流

## 目标

约束开发者或 AI 在实际改代码时的执行顺序、实现边界和落点选择，减少误改、越界改动和无效返工。

## 改代码前先做什么

- 先读取当前模块相关约束文档，再动手。
- 先确认问题属于页面表现、流程行为、状态管理、模型映射还是后端接入。
- 先确认改动应该落在哪一层，不要一上来就在页面里堆业务逻辑。
- 如果用户刚刚提到某个文件可能被改过，先重新读取当前文件内容。

## 推荐执行顺序

### 1. 先确认事实来源

- 产品规则先看产品文档。
- 架构和边界先看架构与设计文档。
- AI 协作规则先看实现、注释、评审、提交等 workflow 文档。

### 2. 先收敛问题位置

- 页面布局问题优先在页面组件和页面本地样式文件或样式目录解决。
- 页面交互和状态切换优先在 `useXxxViewModel` 解决。
- DTO 到页面模型映射优先在 `transformers/` 解决。
- mock / backend 调用差异优先在 `services/` 解决。
- 流程中间态和跨页共享状态优先在 `repository.ts` 解决。

### 3. 再做最小改动实现

- 优先做能满足当前目标的最小修改。
- 不顺手重构无关模块。
- 不在一个提交里混入不相关行为调整。

### 4. 最后补验证与注释

- 关键改动后补上必要注释。
- 检查当前模块是否出现新的类型或 lint 问题。
- 如果改动涉及异常路径，至少人工确认代码里已有降级分支。

## 最小验证清单

- 页面可直接打开时，主流程是否仍正常。
- 缺参、脏参数或上下文缺失时，是否存在明确回退路径。
- 空态、错误态、失败态是否仍完整。
- 返回路径、关闭路径或重新进入路径是否仍然闭环。
- 需要重试的流程是否仍能重试，而不是只能刷新页面。
- 页面重新进入、状态恢复或二次打开时，行为是否与首次进入一致。

## 页面组件规则

- 页面组件负责布局、组装、展示和配置对象拼装。
- 页面中的状态、提交动作、导出逻辑、弹窗开关、路由跳转、流程控制等逻辑部分，优先落入 ViewModel。
- 组件文件仍应把复杂业务判断尽量下沉到 ViewModel、transformer、service 或 repository，而不是继续堆在 JSX 中。

## 组件文件组织规则

- 导出的主组件或主 hook 优先放在文件前部，让读者打开文件先看到对外入口。
- 只服务当前文件的辅助函数、格式化函数、分流函数放在主入口后面，避免文件开头被实现细节淹没。
- 辅助函数优先使用 `function` 声明，不使用箭头函数，便于利用变量提升理解调用链。
- barrel file 不是默认要求。文件数量不多时，直接相对路径导入通常更清晰。

## 命名规则

- 面向 UI 事件的处理函数统一使用 `onXxx` 命名，例如 `onSubmit`、`onConfirm`、`onTabChange`。
- 传递给组件的事件回调 prop 也统一使用 `onXxx` 命名，避免同一项目里同时出现 `onSubmit`、`submitHandler`、`handleSubmit` 这类漂移写法。
- 非事件性质的纯函数、数据映射函数、请求函数、状态收敛函数，不使用 `on` 前缀。
- 函数、变量、组件、hook、类型命名应直接表达职责和语义，避免使用 `data`、`info`、`temp`、`handler`、`manager`、`commonFn` 这类空泛名字。
- 命名优先表达“它服务什么对象、解决什么问题、在什么场景被消费”，不要只表达动作，不表达对象。

## 复用搜索顺序

- 新增纯函数、局部工具方法或辅助组件前，先看当前文件和当前页面目录是否已有可直接复用的实现。
- 再看当前模块内的近邻目录，例如 `components/`、`view-model/`、`utils/`、`services/`。
- 只有当需求已经明显具备跨页面或跨模块复用价值时，再扩大到模块级共享目录或项目级共享目录。
- 不默认要求 AI 或开发者在每次改动前全项目遍历搜索；搜索范围应先近后远，避免为了一次局部改动制造过高成本。
- 如果只是当前页面私有逻辑，优先留在页面附近，不因为“以后可能复用”就立即上提到高层级 `utils/` 或 `common/`。

## 页面目录实现约束

- 当页面入口同时承载多个状态分支、多个大区块和较多局部渲染逻辑时，应优先把页面升级成局部模块结构，而不是继续把实现压在单个入口文件里。
- 页面局部结构变多时，优先拆 `components/` 和 `view-model/`，而不是继续往 `index.tsx` 里堆实现细节。
- 页面样式实现不绑定 `style.ts` 这一命名；重点是页面样式与页面边界保持一致。
- 页面目录内的函数和文件应尽量保持单一职责；如果一个入口同时承载事件响应、数据转换、请求组装、状态收敛多个职责，应优先拆分。

## ViewModel 规则

- 只要逻辑仍然服务于当前页面，就优先先放入页面级 ViewModel，再决定是否需要继续下沉到 transformer、service 或 repository。
- ViewModel 内部只要出现可独立入参、可单独测试、与 React 生命周期无关的逻辑，就优先提取出来，并使用 `function` 声明。
- 上述可提取逻辑分为两类：
  - 纯函数，例如文件名拼接、默认值回退、展示文案收敛。
  - 虽包含副作用、但已不依赖 React hook 上下文、可通过参数独立调用的辅助逻辑，例如打印流程编排、DOM 清理、浏览器事件绑定。
- 不把纯逻辑或已可参数化的副作用辅助逻辑长时间内联在主 hook 主体中。
- ViewModel 文件本身过长时，优先提取 `helpers.ts` 或局部目录，而不是继续内联辅助逻辑。

## 纯逻辑文件规则

- 纯逻辑文件适用于不直接承载 JSX 的文件，例如 `transformers/`、`services/`、`repository.ts`、工具函数文件等。
- 当文件同时承载多段独立逻辑，且继续平铺会明显降低可读性时，优先按职责拆分成多个文件；如果暂时不拆，再考虑用清晰区块辅助阅读。
- 纯逻辑文件可以使用 `// #region <中文名称>` 组织职责区块，但 region 只能辅助阅读，不能替代合理拆分。
- region 更适合用于 transformer、service、repository、复杂工具函数文件，不建议把它当成组件 TSX 文件的默认组织方式。
- 私有辅助函数优先使用 `function` 声明，不使用箭头函数，减少阅读时的声明跳转。

## 长文件治理规则

- 不把“补更多注释”或“再补一个 region”当作长文件问题的默认解法。
- 页面组件过长时，优先拆出子组件、配置文件、样式实现文件、hook 或同目录辅助文件。
- 纯逻辑文件过长时，优先拆出 DTO 映射、状态收敛、存储读写、工具函数等职责清晰的独立文件。
- 当单文件已经明显影响通读效率，应优先拆结构，而不是继续堆注释和区块。
- 单一职责的判断标准不是“文件越小越好”，而是“当前文件或函数是否还能被一眼看出主职责”。

## 实现边界

- 先理解为什么现有代码这样分层，再决定把改动放在哪。
- 如果某个产品规则仍未确认，应先采用最小 fallback，而不是擅自扩展复杂方案。
- 如果发现问题不在当前模块，应明确指出边界，不要无声越界修改。
