# 实现工作流

## 目标

约束开发者或 AI 在实际改代码时的执行顺序、实现边界和落点选择，减少误改、越界改动和无效返工。

## 改代码前先做什么

- 先读取当前模块相关约束文档，再动手。
- 先确认问题属于页面表现、流程行为、状态管理、模型映射还是后端接入。
- 先确认改动应该落在哪一层，不要一上来就在页面里堆业务逻辑。
- 如果用户刚刚提到某个文件可能被改过，先重新读取当前文件内容。

## 推荐执行顺序

### 1. 先确认事实来源

- 产品规则先看产品文档。
- 架构和边界先看架构与设计文档。
- AI 协作规则先看实现、注释、评审、提交等 workflow 文档。

### 2. 先收敛问题位置

- 页面布局问题优先在页面组件和 `style.ts` 解决。
- 页面交互和状态切换优先在 `useXxxViewModel` 解决。
- DTO 到页面模型映射优先在 `transformers/` 解决。
- mock / backend 调用差异优先在 `services/` 解决。
- 流程中间态和跨页共享状态优先在 `repository.ts` 解决。

### 3. 再做最小改动实现

- 优先做能满足当前目标的最小修改。
- 不顺手重构无关模块。
- 不在一个提交里混入不相关行为调整。

### 4. 最后补验证与注释

- 关键改动后补上必要注释。
- 检查当前模块是否出现新的类型或 lint 问题。
- 如果改动涉及异常路径，至少人工确认代码里已有降级分支。

## 页面组件规则

- 页面组件负责布局、组装、展示和配置对象拼装。
- 页面中的状态、提交动作、导出逻辑、弹窗开关、路由跳转、流程控制等逻辑部分，优先落入 ViewModel。
- 组件文件仍应把复杂业务判断尽量下沉到 ViewModel、transformer、service 或 repository，而不是继续堆在 JSX 中。

## ViewModel 规则

- 只要逻辑仍然服务于当前页面，就优先先放入页面级 ViewModel，再决定是否需要继续下沉到 transformer、service 或 repository。
- ViewModel 内部只要出现可独立入参、可单独测试、与 React 生命周期无关的逻辑，就优先提取出来，并使用 `function` 声明。
- 上述可提取逻辑分为两类：
  - 纯函数，例如文件名拼接、默认值回退、展示文案收敛。
  - 虽包含副作用、但已不依赖 React hook 上下文、可通过参数独立调用的辅助逻辑，例如打印流程编排、DOM 清理、浏览器事件绑定。
- 不把纯逻辑或已可参数化的副作用辅助逻辑长时间内联在主 hook 主体中。

## 纯逻辑文件规则

- 纯逻辑文件适用于不直接承载 JSX 的文件，例如 `transformers/`、`services/`、`repository.ts`、工具函数文件等。
- 当文件同时承载多段独立逻辑，且继续平铺会明显降低可读性时，优先按职责拆分成多个文件；如果暂时不拆，再考虑用清晰区块辅助阅读。
- 私有辅助函数优先使用 `function` 声明，不使用箭头函数，减少阅读时的声明跳转。

## 实现边界

- 先理解为什么现有代码这样分层，再决定把改动放在哪。
- 如果某个产品规则仍未确认，应先采用最小 fallback，而不是擅自扩展复杂方案。
- 如果发现问题不在当前模块，应明确指出边界，不要无声越界修改。
