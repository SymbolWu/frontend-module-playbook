# 提交工作流

## 目标

约束开发者或 AI 在代为执行 git 提交时的行为和提交信息风格，保证提交记录易读、可追溯。

## 提交前检查

- 先确认本次改动范围是否仍然聚焦于当前任务。
- 先确认没有误带入无关文件。
- 如果用户刚刚修改过文件，提交前重新读取当前文件，避免覆盖或误解最新内容。
- 如果存在与当前任务无关的脏改动，不要擅自一并提交。

## 提交信息约束

- 推荐格式：`<type>: <简短描述>`。
- 常用类型优先使用：`feat:`、`fix:`、`refactor:`、`docs:`、`chore:`、`test:`。
- 描述要直接说明动作和对象，不写空泛表述。
- 一次提交只表达一个主要意图。
- 如果改动明确属于某个页面或层级，描述中建议带上模块或页面对象。

## 推荐写法

- `feat: 新增用户列表筛选面板`
- `fix: 修复报表页返回跳转逻辑`
- `docs: 补充模块实现说明`
- `refactor: 调整报告页视图模型结构`

## 不推荐写法

- `update code`
- `modify something`
- `处理问题`
- `fix bug`

## 提交动作约束

- 默认只提交本次任务直接相关的文件。
- 不因为顺手方便而把注释、功能、配置、格式化大杂烩进同一个提交。
- 如果本次任务主要是文档或注释，应优先使用 `docs:`，除非它同时改变了实际行为。

## AI 额外要求

- 在建议提交信息时，优先输出简短、清晰的版本。
- 在执行提交前，应尽量理解本次改动的主线，不要只按文件名机械拼接提交信息。
- 如果当前工作区仍有未解决风险，应先告知，再决定是否提交。
