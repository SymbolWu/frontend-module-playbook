# 提交工作流

## 目标

约束开发者或 AI 在代为执行 git 提交时的行为和提交信息风格，保证提交记录易读、可追溯。

## 提交类型

默认使用以下类型前缀：

- `NEW:` 新增全新能力、模块、文档或结构
- `IMPROVE:` 对现有实现或结构做增强、重构、优化
- `FIX:` 修复错误、行为问题或错误说明
- `DOC:` 纯文档、注释、说明类调整
- `RELEASE:` 版本发布、release note、版本信息更新
- `TEST:` 测试、验证、校验相关改动
- `BREAKING:` 破坏性变更，影响旧使用方式或旧结构

## 提交前检查

- 先确认本次改动范围是否仍然聚焦于当前任务。
- 先确认没有误带入无关文件。
- 如果用户刚刚修改过文件，提交前重新读取当前文件，避免覆盖或误解最新内容。
- 如果存在与当前任务无关的脏改动，不要擅自一并提交。

## 提交信息约束

- 推荐格式：`<TYPE>: <简短描述>`。
- 类型命名应使用大写前缀，并与团队约定保持一致。
- 类型前缀可以使用英文大写，但冒号后的描述应使用中文。
- 描述要直接说明动作和对象，不写空泛表述。
- 一次提交只表达一个主要意图。
- 如果改动明确属于某个页面或层级，描述中建议带上模块或页面对象。

## 推荐写法

- `NEW: 新增用户列表筛选面板`
- `FIX: 修复报表页返回跳转逻辑`
- `DOC: 补充模块实现说明`
- `IMPROVE: 调整报告页视图模型结构`
- `RELEASE: 发布 v0.1.0`
- `BREAKING: 删除旧的仓库接入路径`

## 不推荐写法

- `update code`
- `modify something`
- `处理问题`
- `fix bug`

## 提交动作约束

- 默认只提交本次任务直接相关的文件。
- 不因为顺手方便而把注释、功能、配置、格式化大杂烩进同一个提交。
- 如果本次任务主要是文档或注释，应优先使用 `DOC:`，除非它同时改变了实际行为。

## 类型选择建议

- 新增全新内容时，用 `NEW:`
- 结构优化、重写、增强时，用 `IMPROVE:`
- 纠正错误时，用 `FIX:`
- 纯说明文档和注释更新时，用 `DOC:`
- 版本发布和 release note 时，用 `RELEASE:`
- 变更会影响旧接入路径、旧目录或旧流程时，用 `BREAKING:`
- 提交标题中的具体动作描述统一使用中文，不在同一仓库内混用中英文描述风格。

## AI 额外要求

- 在建议提交信息时，优先输出简短、清晰的版本。
- 在执行提交前，应尽量理解本次改动的主线，不要只按文件名机械拼接提交信息。
- 如果当前工作区仍有未解决风险，应先告知，再决定是否提交。
