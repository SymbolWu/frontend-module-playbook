# 文档整理工作流

## 目标

约束开发者或 AI 在整理接口文档、确认清单、需求澄清和实现说明时的工作方式，保证文档结构清晰、信息可追溯、方便后续继续协作。

## 整理前先做什么

- 先区分这是“原始资料转写”还是“基于代码现状的分析整理”。
- 先确认文档的目标读者：产品、前端、后端、AI 接手者，还是混合读者。
- 先确认哪些内容是事实、哪些内容是差异、哪些内容是待确认项，不要混写。
- 如果文档基于代码现状整理，先回到代码和现有文档核对，不要只凭对话记忆写结论。

## 文档结构原则

- 优先按“背景 -> 当前现状 -> 差异 / 风险 -> 待确认项 -> 结论 / 下一步”组织。
- 接口文档优先按“基础信息 -> 主流程 -> 各接口 -> 关键字段 -> 前端真正关心的数据”组织。
- 确认清单优先按“问题 -> 当前理解 -> 需要人类确认的点”组织。
- 需求澄清文档优先写成可直接回复的问答格式，而不是散乱备忘录。

## 写作要求

- 标题要直接表达用途，不写空泛题目。
- 段落优先服务快速理解，不要把一堆上下文揉成超长段。
- 能枚举的内容就枚举，尤其是接口字段、差异项、确认项、风险点。
- 对关键字段名、接口路径、状态枚举要保持与代码或接口中的原文一致。

## 差异分析文档要求

- 先分别列“当前前端现状”和“目标接口 / 产品规则”。
- 再写“差异结论”，明确是字段不一致、流程不一致、状态语义不一致还是展示模型不一致。
- 差异分析的目标是帮助人做决策，不是直接输出一大段改造代码。

## 不推荐写法

- 直接堆原始资料，不做分层。
- 把事实、推测、建议写在同一段里。
- 把待确认项写成模糊问题，导致人类无法直接回答。

## 推荐写法

- 明确写当前实现如何工作。
- 明确写与目标规则相比差在哪。
- 明确写这个差异会影响哪个页面、哪个字段、哪段流程。
- 明确写这个问题需要人类确认什么。
