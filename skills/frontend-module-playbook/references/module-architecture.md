# 前端模块架构指南

## 目标

让一个页面级业务模块在继续演进时，仍然保持职责清晰、文件边界稳定、便于 AI 和人类共同维护。

这份文档的重点不是限定某一种技术栈，而是提供一套更通用的前端模块分层与目录设计建议。

## 基本原则

- 以可维护性优先，不追求过度抽象。
- 以职责清晰优先，不把页面、状态、数据映射、请求协议和共享中间态混在一起。
- 以模块内高内聚优先，减少无必要的跨目录分散。
- 以渐进演进优先，不在文件很少时提前搭过重骨架。
- 以单一职责优先，不让一个文件、一个目录或一个函数同时承担过多不同性质的工作。
- 以模块化优先，让页面私有能力、模块级共享能力、项目级公共能力逐层分开。

## 为什么需要分层与目录设计

如果一个前端模块没有稳定的分层与目录设计，常见问题会很快出现：

- 页面组件既负责布局，又负责流程判断、状态编排和请求拼装。
- DTO、领域数据和页面展示结构混在一起，导致接后端或改 mock 时牵一发而动全身。
- 页面逻辑被临时函数和局部状态不断吞噬，最后只能靠更多注释维持可读性。
- 组件、样式、类型、工具函数散落在多个目录，后续接手者很难判断边界。

目录设计不是为了“看起来规范”，而是为了在模块变复杂后，仍然能快速判断：

- 这段逻辑应该落在哪一层。
- 当前问题该从哪个目录排查。
- 未来要复用时该上提，还是继续留在页面内。

## 通用前端目录结构建议

适用于中小型前端业务模块的基础结构：

```text
FeatureModule/
  api/
  components/
  constants/
  hooks/
  models/
  pages/
  repositories/
  services/
  styles/
  transformers/
  utils/
```

说明：

- `api/` 适合放接口定义、请求路径常量或协议描述；如果项目约定直接由 `services/` 承载，也可以不单独拆出。
- `components/` 放模块内多个页面共享的展示组件；只在单页使用的组件优先留在页面目录内。
- `constants/` 放状态枚举、默认配置、常量映射等稳定值。
- `hooks/` 放模块级复用 hooks；只服务单页的 hook 优先放在页面目录下。
- `models/` 放 DTO、领域模型、页面模型、类型定义等。
- `pages/` 放页面入口与页面内局部结构。
- `repositories/` 放跨页共享的中间态、缓存态、流程恢复态等。
- `services/` 放请求、mock/backend 适配、底层协议处理。
- `styles/` 放模块级公共样式资源；如果样式主要按页面内聚，也可以保持很轻甚至没有该目录。
- `transformers/` 放 Source Data、Domain Model、View Model 之间的转换逻辑。
- `utils/` 放真正具备模块级复用价值的工具函数；不要把页面私有函数过早抽成 utils。

## 模块级目录与页面级目录如何配合

推荐思路是：

- 模块级目录承载“跨页面共享能力”。
- 页面级目录承载“只服务当前页面的结构与逻辑”。

模块级目录适合放：

- 模块内多个页面共享的模型、状态枚举、请求方法、转换层、共享组件。

页面级目录适合放：

- 当前页面入口。
- 当前页面样式。
- 当前页面局部组件。
- 当前页面 ViewModel。
- 只服务当前页面的辅助逻辑。

不要一开始就把所有逻辑抽到模块级目录。页面专属逻辑先留在页面附近，只有在明确出现跨页面复用价值后，再评估上提。

## 页面级目录结构建议

适用于单个页面逐步复杂后的页面级结构：

```text
SomePage/
  index.tsx
  components/
  view-model/
  styles/
```

也可以根据项目习惯落成：

```text
SomePage/
  index.tsx
  style.ts
  components/
  view-model/
```

重点不是文件名必须叫 `style.ts`，而是页面样式、页面组件和页面状态组织要保持边界一致。

## 页面目录组织

- `index.tsx` 负责页面入口编排、主要分支判断和大区块组装。
- `components/` 只放当前页面内部使用的展示区块组件。
- `view-model/` 只放当前页面的主 hook 与只服务该 hook 的辅助逻辑。
- `styles/`、`style.ts`、`index.module.scss`、`index.less` 等样式实现文件都可以存在，关键是样式与页面保持内聚。

## 样式组织原则

当前 skill 不绑定 `styled-components`、Less、Sass 或 CSS Modules 中的任一方案。

这里强调的是样式组织原则，而不是具体文件名：

- 一个页面模块或组件模块通常应有与其边界一致的样式实现文件或样式目录。
- 模块级公共样式应有单独归档位置，不要混在页面私有样式中。
- 页面私有样式优先跟页面目录共置，减少样式来源分散。
- 公共样式只有在确实被多个页面或多个模块复用时才上提。
- 不要在 JSX 中长期堆叠大量内联样式或魔法数。

可接受的样式落地形式示例：

- `style.ts`
- `styles.ts`
- `index.less`
- `index.module.scss`
- `styles/`

Tailwind 这类原子化方案不在这份文档重点讨论范围内，但它同样不改变“样式需要按模块边界组织”的原则。

## 什么时候把页面升级成“小模块”

当页面入口文件同时具备以下特征时，应考虑把页面升级成 `index.tsx + components/ + view-model/ + 样式文件或样式目录` 的局部模块结构：

- 同时承载加载态、空态、内容态多个分支。
- 页面中有 2 到 3 个以上大展示区块。
- 存在多段局部渲染函数、格式化函数或事件处理流程。
- 页面状态与流程控制已经明显超过纯展示组件负荷。

## 公共目录什么时候抽，什么时候不要抽

优先抽到模块级目录的情况：

- 被多个页面复用。
- 职责已经稳定。
- 抽离后不会失去业务语义。

不建议过早抽离的情况：

- 只服务当前页面。
- 规则还在快速变化。
- 抽离后文件名会变得空泛，例如 `helper.ts`、`utils.ts`、`common.ts`。

判断标准不是“能不能复用”，而是“现在抽离是否真的让边界更清楚”。

## 复用搜索顺序与公共能力上提原则

- 需要新增纯函数、公共方法、局部组件时，优先先看当前页面目录和当前模块内是否已有可复用实现，不默认全项目穷举搜索。
- 搜索顺序优先按“当前文件 -> 当前页面目录 -> 当前模块共享目录 -> 项目级共享目录”递进，而不是一开始就把所有模块扫一遍。
- 页面私有能力默认先留在页面附近；只有在明确出现跨页面复用后，再评估是否上提到模块级目录。
- 模块级能力只有在复用范围继续扩大、并且职责已经稳定时，再评估是否上提到项目级共享目录。
- 公共能力上提的前提不是“未来可能复用”，而是“已经有明确复用证据，且上提后边界更清楚”。
- 如果上提后只能得到 `common.ts`、`utils.ts`、`sharedHelper.ts` 这类空泛命名，通常说明时机还不成熟。

## 模块化与单一职责原则

- 模块化的目标不是把目录拆得越多越好，而是让页面、状态、数据映射、请求适配、共享中间态各自留在稳定边界内。
- 单一职责的判断标准不是文件长度，而是当前目录、文件或函数的主职责是否明确且稳定。
- 如果一个函数同时承担事件响应、数据转换、请求组装、状态收敛多个职责，应优先拆开，而不是继续堆在一个入口里。
- 如果一个页面目录同时堆放展示组件、状态流程、纯工具函数、样式和请求拼装，且边界已经模糊，应优先升级为更清晰的局部模块结构。
- 不要把“尽量复用”理解成“尽量公共化”；真正稳妥的做法是先局部内聚，复用证据充分后再上提。

## 分层约束

### 页面组件

- 负责布局、组装、展示和配置对象拼装。
- 不直接塞复杂业务判断。
- 不持续追加状态、流程控制、导出逻辑、提交流程等行为实现。

### ViewModel

- 负责页面状态、提交动作、跳转时机、轮询和错误处理。
- 页面中的状态与行为逻辑默认优先沉淀在 ViewModel 中。
- 页面专属逻辑优先留在页面目录下，不默认抽到模块级公共目录。

### Transformer

- 负责 DTO、原始数据、领域模型、页面模型之间的转换。
- 统一承接展示字段派生、状态映射和默认值回退。
- 不把页面跳转和 UI 状态写进 transformer。

### Service

- 负责请求、mock/backend 适配和底层协议处理。
- 不在 service 里直接写页面表现逻辑。

### Repository

- 负责模块内需要跨页保留的中间态。
- 不把纯页面瞬时状态塞进 repository。

## 组件拆分规则

- 当页面内部展示区块达到 2 到 3 个以上，且每个区块都有独立 props、独立渲染细节时，优先拆到 `components/`。
- 内部组件命名应直接表达业务区块，例如“顶部摘要区”“规则区”“导出弹窗”，而不是空泛地命名为 `PanelA`、`SectionB`。
- 组件、hook、helpers 文件命名也应直接表达职责和消费场景，不使用 `common`、`temp`、`manager` 这类语义模糊的名字。
- 页面内部组件数量不多时，不强制增加 `components/index.ts` 一类 barrel file，直接相对路径导入通常更清晰。

## ViewModel 进一步拆分规则

- 如果主 hook 已经同时包含主流程和多段只服务该 hook 的辅助逻辑，可进一步收敛为 `view-model/index.ts` + `view-model/helpers.ts`。
- `view-model/helpers.ts` 只服务当前页面状态编排，不应自然扩展成跨页面工具库。
- 一旦出现跨页面复用需求，再评估是否下沉到更高层目录。

## 长文件处理原则

- 不把“补更多注释”当作长文件问题的默认解法。
- 页面组件过长时，优先拆出子组件、局部渲染组件、配置文件、hook、样式文件或同目录辅助文件。
- 纯逻辑文件过长时，优先拆出状态收敛、工具函数、映射逻辑、存储读写等独立文件。
- 拆分后的入口文件应仍然保留主流程，做到“打开入口先看懂流程，深入时再进入细节文件”。
